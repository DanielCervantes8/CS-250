#CS-250 Portfolio Reflection
#Sprint Review and Retrospective Artifact

This repository includes my Sprint Review and Retrospective from the SNHU Travel Scrum-Agile pilot project. In this project, 
I served as Scrum Master and helped guide the team through transitioning from a traditional waterfall approach to an Agile 
Scrum framework. This artifact demonstrates how I applied Agile principles, facilitated collaboration, and supported 
incremental delivery.

#How do I interpret user needs and implement them into a program? How does creating user stories help with this?

I interpret user needs by breaking large requirements into smaller, clear, actionable user stories that focus on delivering 
value. Instead of thinking in terms of technical tasks first, I focus on what the user wants to accomplish and why it 
matters. Creating user stories helps clarify acceptance criteria before development begins, which reduces ambiguity and 
prevents rework later.

During the SNHU Travel project, user stories allowed us to adapt when priorities shifted toward detox and wellness travel 
without restarting development. Because stories were small and iterative, we could reprioritize the backlog and continue 
delivering working increments. This experience reinforced that structured user stories translate business needs into 
manageable development work while keeping the end user at the center of the process.

#How do I approach developing programs? What Agile processes do I hope to incorporate into my future development work?

I approach development iteratively rather than trying to complete everything at once. Breaking work into sprints creates 
focus and accountability. I prefer defining a clear “Definition of Done” so that each increment is fully tested, reviewed, 
and functional before moving forward.

In the future, I plan to incorporate Agile practices such as sprint planning with realistic workload commitments, daily 
stand-ups for transparency and blocker resolution, sprint reviews to validate value delivery retrospectives for continuous 
improvement

The biggest lesson I learned is that adaptability is more powerful than rigid planning. Agile allows teams to respond to 
change without losing progress. That flexibility will be essential in real-world software environments where requirements 
evolve.
